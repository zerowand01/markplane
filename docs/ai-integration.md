# AI Integration Guide

Markplane is designed from the ground up for AI consumption. Every structural decision — small focused files, index-based routing, generated context summaries — optimizes for LLM context windows.

## The .context/ Directory

The `.context/` directory contains generated summaries that compress project state into token-efficient formats. These files are created and updated by `markplane sync` or `markplane context`.

### summary.md

High-level project overview targeting ~1000 tokens. Contains:

- **Active Epics**: Each epic with completion percentage and remaining items
- **In-Progress Work**: Items currently being worked on, with priority and assignee
- **Blocked Items**: Items with unresolved dependencies
- **Recent Completions**: Items finished within the configured `recent_days` window (default: 7 days)
- **Priority Queue**: Next 5 items in planned/backlog status, sorted by priority
- **Key Metrics**: Total open items broken down by priority, last sync date

This is the single most important file for AI agents to read first.

### active-work.md

Detailed view of items currently in-progress. For each item, includes:

- ID, title, priority, type, effort
- Assignee, epic, and linked plan
- Dependency and blocker relationships

Also lists in-progress plans with their `implements` links.

### blocked-items.md

Items that cannot proceed because they depend on other items that are not yet done. For each blocked item, shows:

- ID, title, priority
- Which specific items are blocking it (with `[[ID]]` cross-references)

### metrics.md

Comprehensive project health data:

- **Status distribution**: Count of items in each status (draft, backlog, planned, in-progress, done, cancelled)
- **Priority distribution**: Count of open items at each priority level
- **Epic progress**: Per-epic completion percentage and item counts
- **Plans summary**: Active vs. completed plans

## INDEX.md Router Pattern

Every directory in `.markplane/` has an `INDEX.md` file that serves as a table of contents and routing guide. This is the key design pattern that makes Markplane efficient for AI consumption.

### How it works

1. An AI agent reads an INDEX.md file (~200 tokens)
2. The index lists all items in that directory with their IDs, titles, and status
3. The agent decides which specific files are relevant to its task
4. The agent loads only those files (each ~200-400 tokens)

This prevents the common anti-pattern of loading an entire directory of files when only a few are relevant.

### Index files

| Index | Contents |
|-------|----------|
| `.markplane/INDEX.md` | Root navigation: links to all directories with item counts |
| `.markplane/roadmap/INDEX.md` | Epics (active, planned, done) with nested task tables and progress |
| `.markplane/backlog/INDEX.md` | Prioritized kanban view (In Progress, Blocked, Planned, Backlog, Drafts, Recently Done) |
| `.markplane/plans/INDEX.md` | All plans with status and linked tasks |
| `.markplane/notes/INDEX.md` | All notes with type and status |
All INDEX.md files (except the initial scaffolded ones) are regenerated by `markplane sync`.

## Token Budget Design

Markplane targets ~1000 tokens for full project state via context files. Individual items are kept under 2000 tokens each.

| Document Type | Target Size | Purpose |
|---------------|-------------|---------|
| summary.md | ~1000 tokens | Full project state overview |
| active-work.md | ~500 tokens | Current sprint focus |
| blocked-items.md | ~200-500 tokens | Items needing attention |
| metrics.md | ~500 tokens | Project health data |
| INDEX.md | ~200-500 tokens | Directory routing |
| Task | ~200-400 tokens | Single work item |
| Epic | ~400-600 tokens | Strategic overview |
| Plan | ~800-2000 tokens | Implementation details |

**Comparison**: A typical monolithic ROADMAP.md + BACKLOG.md can consume 10,000-30,000 tokens with low signal-to-noise. Markplane's equivalent project state fits in ~1000 tokens.

## How AI Coding Tools Should Use Markplane

### Recommended reading order

1. **Read `summary.md`** for a project overview (~1000 tokens). This tells the agent what's active, what's blocked, and what's next.

2. **Read the relevant `INDEX.md`** for item listings. If working on a task, read `.markplane/backlog/INDEX.md`. If exploring epics, read `.markplane/roadmap/INDEX.md`.

3. **Load specific items** as needed. Use the IDs from the index to read individual files: `TASK-042.md`, `PLAN-012.md`, etc.

4. **Use MCP tools** for structured access. Instead of reading raw files, AI tools with MCP support can call `markplane_query`, `markplane_show`, `markplane_context`, etc. See [MCP Setup Guide](mcp-setup.md).

### Example: AI working on a task

```
1. markplane_summary          -> understand project state
2. markplane_query             -> find relevant tasks
   { status: ["in-progress"], priority: ["critical", "high"] }
3. markplane_show              -> read full details of the target item
   { id: "TASK-042" }
4. markplane_context           -> get focused context including dependencies
   { item: "TASK-042" }
5. (implement the feature)
6. markplane_done              -> mark the item complete
   { id: "TASK-042" }
7. markplane_sync              -> regenerate indexes and context files
```

### Example: AI creating a new item

The create-then-edit workflow uses `markplane_add` to scaffold the item, then `markplane_write` to fill in the body:

```
1. markplane_add               -> create the item (scaffolded from template)
   { title: "Add caching layer", priority: "high" }
   returns: { id: "TASK-043" }
2. markplane_write             -> fill in the markdown body content
   { id: "TASK-043", body: "# Add caching layer\n\n## Description\n\n..." }
3. markplane_update            -> move from draft to backlog once fully defined
   { id: "TASK-043", status: "backlog" }
```

Total context consumed: ~2000-3000 tokens for the project management layer, leaving the rest of the context window for code.

### Cross-references

Items reference each other using `[[ID]]` wiki-style syntax. The ID prefix determines the item type and directory:

| Prefix | Type | Directory |
|--------|------|-----------|
| `EPIC` | Epic | `roadmap/` |
| `TASK` | Task | `backlog/` |
| `PLAN` | Plan | `plans/` |
| `NOTE` | Note | `notes/` |

For example, `[[TASK-042]]` resolves to `.markplane/backlog/items/TASK-042.md`.

## The Sync Command

`markplane sync` regenerates all auto-generated files:

1. **INDEX.md files** for every directory (roadmap, backlog, plans, notes, and root)
2. **All .context/ files**: summary.md, active-work.md, blocked-items.md, metrics.md

### When to run sync

- After creating, updating, or archiving items
- After manually editing frontmatter fields
- Before an AI agent reads project state (to ensure context files are current)
- The MCP tool `markplane_sync` provides the same functionality for AI tools

### What sync does NOT do

- It does not modify item files (only reads them)
- It does not change item statuses or metadata
- It does not create or delete items

Sync is always safe to run. It only overwrites generated files (marked with `<!-- Generated by markplane sync -->`).

## CLAUDE.md Integration

Run `markplane claude-md` to generate a snippet for your project's CLAUDE.md file, pointing AI agents to the right entry points:

```markdown
## Project Management
This project uses Markplane for project management. Key files:
- `.markplane/INDEX.md` - Navigation entry point
- `.markplane/.context/summary.md` - Current project state
- `.markplane/backlog/INDEX.md` - All work items
- `.markplane/plans/INDEX.md` - Implementation plans
When working on a task, read the relevant task and its linked plan first.
```
