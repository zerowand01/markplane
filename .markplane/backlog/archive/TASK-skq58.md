# Git-safe architecture: random IDs, gitignore derived files, auto-sync strategy

## Description

Markplane's current architecture has three issues that make it unsafe for git-based team workflows:

1. **ID collisions across branches.** Sequential IDs (`TASK-042`) are allocated from a shared counter in `config.yaml`. When two people create items on separate branches, both increment from the same base state, producing duplicate IDs. On merge: two different files with the same ID, conflicting counter values, and ambiguous cross-references. Sequential IDs require a single writer — git branches are explicitly multiple writers diverging from shared state. These properties are fundamentally incompatible.

2. **Generated file merge conflicts.** `markplane sync` regenerates INDEX.md files (5 total) and `.context/` summaries (4 files). These are fully derived from source files (frontmatter + body). If two branches both run sync, both modify these files — guaranteed merge conflicts even when the underlying data merges cleanly. Since these files are derived, the conflicts are always trivially resolved by re-running sync, but they create noise and friction.

3. **Sync modifies source files.** `normalize_positions()` is currently bundled into `sync_all()`. Unlike INDEX and `.context/` generation, it modifies task frontmatter (source files) to rewrite fractional position keys into clean sequential ones. This means sync creates real git changes, making auto-sync unsafe — it would dirty the working tree with source file modifications.

### Solution: Random IDs

Replace sequential counter-based IDs with short random IDs. Format: `TASK-k7x9m` — 5 lowercase alphanumeric characters from a reduced 32-character alphabet (a-z, 0-9 minus ambiguous chars `0, o, 1, l`). This gives ~33 million combinations per prefix — collision probability is negligible for any real project.

- No coordination required — fully offline, works across branches
- No counter in `config.yaml` — the `counters` map is removed entirely
- IDs are stable, short (10 chars total), and typeable
- Chronological ordering comes from frontmatter `created` date, not from the ID
- All existing items (including markplane's own `.markplane/` data) are migrated to the new ID format

### Solution: Gitignore derived files

Add INDEX.md (all 5) and `.context/` (entire directory) to `.gitignore` within `.markplane/`. These are fully derived from source files and can be regenerated instantly by `markplane sync`. Gitignoring them eliminates merge conflicts on derived files and makes auto-sync safe — regenerating derived files never pollutes `git status`.

The trade-off is that these files won't be visible on GitHub/GitLab or in a fresh clone until sync runs. This is acceptable because: (1) the MCP server and web UI read source files directly via API for most operations, (2) `markplane init` and server startup can auto-generate them, (3) AI agents interact through MCP tools, not by browsing raw files.

### Solution: Auto-sync strategy and normalize separation

With derived files gitignored, auto-sync becomes safe. Sync should run at these points:

- **`markplane init`** — project is immediately usable after scaffolding
- **`markplane sync`** — explicit manual command, always available
- **`markplane mcp` startup** — AI agent gets current context on connect
- **`markplane serve` startup** — dashboard shows fresh context immediately
- **After API mutations (server mode)** — debounced 2 seconds. Multiple rapid mutations (drag-and-drop reordering) coalesce into one sync
- **After external file changes (server mode)** — debounced 10 seconds. Longer window because external edits (AI agents editing multiple files) are bursty
- **NOT on every CLI command** — keep CLI fast, user runs `markplane sync` explicitly

Debounce resets on each new event, so sync runs after activity settles, not during it.

**Separate `normalize_positions` from sync.** Normalization rewrites source files (task frontmatter position keys) and should not run automatically. It becomes a separate explicit command (`markplane sync --normalize` or `markplane normalize`) for occasional housekeeping. The fractional keys generated by drag-and-drop (`a0V`) work correctly for ordering — normalization to clean sequential keys (`a0`, `a1`, `a2`) is cosmetic.

Files likely affected:

- `crates/markplane-core/src/models.rs` — ID validation regex update
- `crates/markplane-core/src/context.rs` — `sync_all()` decouple normalize
- `crates/markplane-core/src/templates.rs` — init template adds `.gitignore`
- `crates/markplane-cli/src/commands/serve.rs` — auto-sync on startup + debounced sync after mutations/file changes
- `crates/markplane-cli/src/mcp/` — auto-sync on MCP server startup
- `crates/markplane-cli/src/commands/sync.rs` — `--normalize` flag
- Integration tests throughout

This is greenfield — no backward compatibility required. A one-off migration script converts existing sequential IDs to the new format. The script takes a path to a `.markplane/` directory, builds an old→new ID mapping, rewrites all frontmatter references and wiki-links, renames files, removes the `counters` map from config.yaml, and runs sync.

**Ordering constraint**: the migration script must run and be committed BEFORE the code changes that update ID validation. Once `next_id()` generates random IDs and validation only accepts the new format, old sequential items would fail validation.
